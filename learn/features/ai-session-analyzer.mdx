## What You'll Accomplish

AI Session Analyzer is Hoop's first AI-powered risk analysis capability for terminal sessions. It evaluates commands before execution and applies configured policies automatically, so you can:

- Detect risky commands in real time before impact
- Block destructive or high-risk operations proactively
- Apply consistent policy decisions across resources
- Give admins centralized control while users work normally

---

## Overview

AI Session Analyzer is a real-time risk analysis agent embedded in the Web Terminal. It uses LLM reasoning plus resource context to identify data security and reliability risks, such as:

- Destructive operations
- Long-running lock risks
- Heavy reads without proper indexing
- Commands that violate organizational policy

This feature introduces session-level controls that admins configure once, while common users consume passively during their sessions.

## Who Configures vs Who Uses

| Role | Experience |
|------|------------|
| **Admin** | Configures provider, model, API key, and rule policies |
| **Common User** | Does not configure anything; receives Allowed/Blocked outcomes in terminal sessions |

## How It Works

<Steps>
  <Step title="Admin Configures AI Provider">
    Configure tab is completed first with provider, model, and API key.
  </Step>
  <Step title="Admin Creates Rules">
    Rules define scope (roles/attributes) and actions (for example, alert or block).
  </Step>
  <Step title="User Runs Terminal Commands">
    During a session, commands are evaluated in real time against configured rules.
  </Step>
  <Step title="System Applies Action">
    Session is marked Allowed or Blocked, with risk status available in terminal and session details.
  </Step>
</Steps>

## Admin Journey

### 1) First Configuration (Required)

The **Configure** tab is the required starting point. AI Session Analyzer will not run until configuration is valid and saved.

If an admin opens the **Rules** tab before completing configuration, the UI shows an empty state.

#### Provider Options

Admins can select one provider at a time:

- Azure OpenAI
- OpenAI
- Claude
- Anthropic
- Custom (OpenAI-compatible API)

**Behavior rules:**

- Only one provider can be active at a time
- Selected provider card shows active highlight and border
- Changing provider resets selected model
- Changing provider clears API key input

#### OpenAI, Claude, Anthropic

- Model is selected from a dropdown
- Admin configures model plus API key

#### Custom (OpenAI-compatible API)

- Admin fills required custom endpoint fields
- Configuration must still include a valid model and API key

#### Validation and Save

- Required fields cannot be empty
- Credentials are validated before saving
- On successful save, show: `Configuration saved.`
- If user leaves with unsaved changes, show a confirmation prompt

### 2) First Rule Creation

After configuration is complete, the rule flow is enabled.

<Steps>
  <Step title="Open Empty State">
    Admin lands on AI Session Analyzer empty state in Rules.
  </Step>
  <Step title="Create First Rule">
    Click to create the first rule.
  </Step>
  <Step title="Define Rule">
    Add Name, optional Description, then configure roles, attributes, and risk actions.
  </Step>
  <Step title="Save and Redirect">
    On success, redirect to Rules list and show success indicator.
  </Step>
</Steps>

**Important:** Recommended actions are preselected by the system. If an admin chooses a different action, the `recommended` tag remains visible.

If user leaves the rule page with unsaved changes, show a confirmation prompt.

### 3) Normal Admin Usage

After first configuration and first rule creation:

- Rules list shows existing rules
- **Create new rule** button is available in the top area
- Admin can create additional rules with the same flow
- Admin can configure existing rules from the list
- Admin can delete rules
- Delete requires confirmation modal
- After deletion, redirect to list and remove the rule
- Leaving unsaved edit pages prompts confirmation

## Common User Experience

Common users do not configure AI Session Analyzer. They only experience its effects during terminal usage.

### In-Terminal Behavior

When a terminal session starts:

- Terminal shows an in-session status card: `AI Session Analyzer is thinking...`
- Analyzer evaluates activity in real time
- If no rule is violated, show Allowed/success status (non-blocking notification)
- If any configured risk rule is triggered, show Blocked status (prominent restriction notification)

### Session Details Review

In the session detailed view:

- Top status list includes **AI Session Analyzer** item
- Item displays action status: **Allowed** or **Blocked**
- A dedicated card shows:
  - Title plus risk level indicator (**Low Risk**, **Medium Risk**, **High Risk**)
  - Action applied by rule (**Allowed** or **Blocked**)

When Tabs are enabled in production, this card should be included in the Tabs structure.

---

## Status Types

| Status | Meaning |
|--------|---------|
| **Allowed** | Session complies with configured risk rules |
| **Blocked** | Session violates one or more configured risk rules |

<Note>
Some sessions may still require Review, even when AI Session Analyzer marks them as Allowed or Blocked.
</Note>

## Troubleshooting

### Rules Tab Is Empty

**Check:**
1. Configure tab was completed first
2. Provider, model, and API key are valid
3. Configuration was saved successfully

### Save Is Blocked

**Check:**
1. All required fields are filled
2. API key is valid for selected provider/model
3. Custom provider fields are complete (if using OpenAI-compatible API)

### Analyzer Not Applying Rules in Session

**Check:**
1. At least one rule exists and is active
2. Rule scope (roles/attributes) matches the current session
3. User is running commands in Web Terminal where analyzer status is shown
