## Requirements
- Azure account
- `API_URL` of the Hoop gateway instance

Contact the hoop gateway administrator to retrieve the `API_URL` address.

## Identity Provider Configuration

<Steps titleSize="h3">
  <Step title="Create an Application">
  	Log in at https://portal.azure.com/
	- Go to **Microsoft Entra ID > App Registration**
	- Pick a name and select Supported account types as needed
	- Register the application with `{API_URL}/api/callback` as the redirect URI

	<Note>
	Use the **single tenant** option for Supported account types. This allows selecting exactly which groups propagate to this application and is required to propagate cloud-only groups as names (see below).
	</Note>

	Ensure default delegated permissions: `User.Read` for sign-in and user profile reads, and `email` to view user email addresses.

	<Info>
	These attributes are created by default. If errors occur, verify these options are set correctly.
	</Info>
  </Step>
  <Step title="Create the Client Credentials">
    - Go to **App Registration > `{AppName}`**
	- In the overview page, copy the **Application (client) ID** value. This is the Client ID.
	- Go to **Certificate & Secrets > Client Secrets > New Client Secret**
	- Copy the Secret Value. This is the Client Secret.
  </Step>
  <Step title="Collect Issuer and Custom Scopes Information">
  The Issuer URL:
  - Go to **App registration > `{AppName}` > Overview**
  - Copy the tenant ID and use it in the value below:
  - `https://login.microsoftonline.com/{tenant_id}/v2.0`

  The Custom Scopes:
  - Use the Client ID to compose the Custom Scopes value
  - `{CLIENT_ID}/.default`

  </Step>
</Steps>

## Configuring Groups Claims

The integration relies on groups propagated in the `id_token`. By default, Azure Entra ID propagates them in the `groups` claim.

**Configure the gateway** to match the claim name of the groups. This ensures groups sync when a user authenticates on Hoop.

<Tip>
  Inspect the claim name being propagated by viewing the gateway logs when a user signs in.
</Tip>

<Note>
  By default the gateway uses the claim name `groups`.
  To change it, go to **Integrations** > **Authentication** and include a distinct group.
</Note>


By default, groups propagate with their object IDs. Managing groups as UUIDs in Hoop can be cumbersome; see the section below to propagate them with their display names instead.

<Warning>
  When you configure groups to sync, you lose admin access on the next sign-in.
  To prevent this, set `ADMIN_USERNAME` on your gateway to change the admin user role to a group associated with your application.
  The admin group name depends on whether you propagate groups as object IDs or as group names (see below).
</Warning>

### Propagating Groups as Names

Propagate groups with their display names for cloud groups:

- Click on **Manifest**
- Edit the `groups` item under `optionalClaims.idToken` and `optionalClaims.accessToken` and add the name `cloud_displayname`
- Set `groupMembershipClaims` to `ApplicationGroup`

```json
(...)
  "groupMembershipClaims": "ApplicationGroup",
  (...)
  "optionalClaims": {
    "idToken": [
      {
        "name": "groups",
        "source": null,
        "essential": false,
        "additionalProperties": [
          "cloud_displayname"
        ]
      }
    ],
    "accessToken": [
      {
        "name": "groups",
        "source": null,
        "essential": false,
        "additionalProperties": [
          "cloud_displayname"
        ]
      }
    ],
    (...)
```

- Click **Save**

This option only works when `groupMembershipClaims` is set to `ApplicationGroup` and groups are assigned to this application.

<Warning>
  Group names are not unique and may cause conflicts during propagation. Object IDs are a safer approach to avoid this problem.
</Warning>

<Info>
  For other group types, you can append additional properties such as `sam_account_name`, `dns_domain_and_sam_account_name`, or `netbios_domain_and_sam_account_name`.
</Info>

### Assign Groups to Application

- Go to **Enterprise Application > `{AppName}` > User and Groups > Add user/group**
- Select the desired groups and click **Select**
- Click **Assign**

<Info>This option requires an AD subscription that supports assigning groups to applications.</Info>

References:

- [Configure Optional Claims](https://learn.microsoft.com/en-us/entra/identity-platform/optional-claims?tabs=manifest#configuring-group-optional-claims)
- [Configure group claims for applications](https://learn.microsoft.com/en-us/entra/identity/hybrid/connect/how-to-connect-fed-group-claims)

## Machine to Machine Access (M2M)

<Warning>Available on version 1.17.14+</Warning>

Access Hoop by issuing access tokens using the OAuth2 Client Credentials Flow. The gateway validates and authenticates any access token generated by the app registered in Azure, provided a service account is active on Hoop. The service account must have the same Object ID as the application, which serves as its main identifier.

<Info>
  Authentication only occurs when a token is issued by the identity provider. The service account resource maps the subject to identify who is accessing the API.
</Info>

### Creating a Service Account

```sh
hoop admin create serviceaccount <azure-app-object-id> \
  --name "My Service Account" \
  --groups admin
```

<Info>
  To obtain the Object ID of the application, navigate to: Azure Portal > Microsoft Entra ID > Enterprise Applications
</Info>

### Creating an Access Token

1. Go to **App Registrations > Certificate & Secrets > New Client Secret**

Copy the secret value and use it as `<APP_CLIENT_SECRET>` in the command below.

2. Generate an access token:

```sh
curl -XPOST -H "Content-Type: application/x-www-form-urlencoded" \
	-d client_id=<APP_CLIENT_ID> \
	-d scope=<APP_CLIENT_ID>/.default \
	-d client_secret=<APP_CLIENT_SECRET> \
	-d grant_type=client_credentials \
	https://login.microsoftonline.com/<TENANT_ID>/oauth2/v2.0/token
```

- APP_CLIENT_ID: the client ID of the application
- APP_CLIENT_SECRET: the client secret generated in step 1
- TENANT_ID: the tenant ID of your app instance

For more information, [see this guide](https://learn.microsoft.com/en-us/entra/identity-platform/v2-oauth2-client-creds-grant-flow#get-a-token).

Access the API:

```sh
export HOOP_TOKEN=eyJ0eXAiOiJKV1QiLCJhb...5Z3Be-kkXkAnAA-zIweYuqEUDA
hoop admin get userinfo
```

<Info>
  If you receive access denied (401), verify that the access token subject matches the subject provided when creating the service account (usually matches the application's object ID).
</Info>
